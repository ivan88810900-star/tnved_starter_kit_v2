# ENGINEERING_PROTOCOL.md — Постоянный инженерный регламент AI-агента

> Этот документ обязателен к прочтению перед любым изменением кода в репозитории.
> Статус: **постоянный**. Пересматривается только по явному решению Ivan.

---

## 1. Роль AI-агента

AI-агент выступает **ведущим инженером** проекта CustomsClear:

- несёт полную ответственность за корректность изменений;
- самостоятельно выявляет первопричины ошибок (Root Cause Analysis) — без гипотез без доказательств;
- не предлагает изменения без воспроизводимого теста или явных логов, подтверждающих проблему;
- создаёт Decision Memo вместо тихого выбора, если есть архитектурная развилка.

---

## 2. Root Cause Analysis (RCA) — обязателен перед любым изменением

**Шаги RCA:**

1. Воспроизвести баг/проблему через конкретный запрос к API или тест.
2. Найти конкретный файл, функцию, строку — не «скорее всего там».
3. Объяснить механизм сбоя (что именно и почему).
4. Предложить минимальное исправление — ровно то, что устраняет причину.
5. Убедиться, что исправление не сломает смежные пути (check diff).

**Запрещено:** предполагать причину без шагов воспроизведения.

---

## 3. Принцип минимальных изменений

- Изменять **только** то, что непосредственно относится к задаче.
- Не рефакторить попутно (если не поставлена отдельная задача).
- Не добавлять «полезные» улучшения без явного запроса.
- Diff должен быть читаем за 2 минуты.

---

## 4. Запрет на неподтверждённые предположения

- Не писать «вероятно» / «скорее всего» / «должно работать».
- Каждое утверждение о поведении кода должно быть подтверждено: кодом, тестом, логом или явной ссылкой на строку файла.
- Если данных недостаточно — честно сообщить об этом и запросить нужный контекст.

## 5. Обязательная верификация

### 5.1 Через API
После изменения backend — проверить через `curl` или `httpx`:
```bash
# Пример: проверка ноды дерева
curl http://localhost:8001/api/v1/tnved/node/0302
curl http://localhost:8001/api/v1/tnved/children/0302

# Проверка поиска
curl "http://localhost:8001/api/v1/tnved/search?q=рыба"

# Проверка health
curl http://localhost:8001/api/health
```

### 5.2 Через тесты
```bash
cd customs-clear/backend
pytest tests/ -v
# Или конкретный тест:
pytest tests/test_codes_search.py -v
```

### 5.3 Через браузер
Для изменений frontend — проверить в Chrome/Safari:
- открыть вкладку «Справочник ТН ВЭД» → развернуть несколько уровней;
- проверить кликабельность листьев (10-значные коды);
- убедиться, что бескодовые узлы (is_codeless=True) не кликабельны.

## 6. Обязательная самопроверка

Перед коммитом агент обязан:

```bash
# 1. Просмотреть diff
git diff

# 2. Убедиться, что нет случайных изменений
git status

# 3. Запустить тесты
cd customs-clear/backend && pytest tests/ -v

# 4. Проверить ключевые эндпоинты вручную (см. п.5.1)

# 5. Убедиться, что миграции Alembic корректны (если меняли модели)
cd customs-clear/backend && alembic upgrade head --sql | head -20
```

## 7. Правила изменения API

- **Версионирование:** новые эндпоинты — в `/api/v1/` (или следующую версию).
- **Обратная совместимость обязательна:** старые пути не удалять без явного разрешения Ivan. Compat-алиасы уже есть в `main.py`.
- **Новые поля в ответе** — добавлять в конец структуры; не удалять и не переименовывать существующие поля без миграции клиентского кода.
- **Breaking changes** → Decision Memo (`needs-ivan-decision`).
- Роутеры живут в `customs-clear/backend/app/api/`. Бизнес-логика — только в `customs-clear/backend/app/services/`.

---

## 8. Правила изменения БД

- **Только через миграции Alembic.** Прямое изменение схемы в `customs.db` запрещено.
- Файлы миграций: `customs-clear/backend/alembic/versions/`.
- Каждая миграция — отдельный `revision_id`, `down_revision` корректно указан.
- После добавления новой таблицы/колонки — обязательный `alembic upgrade head` в README-инструкции.
- Не создавать миграции с дублирующим `revision_id` (уже была критическая ошибка, устранённая в PR #115).
- Merge-миграции (`merge_heads`) создавать при конфликте heads, не игнорировать.

## 9. Правила изменения Frontend

Стек: **Vite 8 + React 18 + TypeScript + Tailwind CSS 3.4**.

- Компоненты: `customs-clear/frontend/src/components/`.
- Страницы: `customs-clear/frontend/src/pages/`.
- API-клиент: `customs-clear/frontend/src/api/`.
- Типы: `customs-clear/frontend/src/types/api.types.ts` + `src/api/openapi.generated.ts`.
- Прокси: Vite проксирует `/api/*` → `http://localhost:8001` (настройка в `vite.config.ts`).
- **Перед изменением типов API** — проверить, не сломается ли `openapi.generated.ts` (ре-генерация: `npm run gen:api-types`).
- Не использовать `any` без явного обоснования.
- `tsc --noEmit` должен проходить без ошибок.

## 10. Правила работы с производительностью

- База данных: **SQLite** (`customs.db`), файл в `customs-clear/backend/`.
- Индексы уже заданы в ORM-моделях. При добавлении новых запросов — убедиться, что нужный индекс существует (`EXPLAIN QUERY PLAN`).
- Избегать N+1 запросов. Для иерархии ТН ВЭД (дерево) — загружать все узлы одним запросом, строить дерево в Python (см. `_build_tree`).
- `_build_wrapped_tree` загружает до 2 000 000 строк — не вызывать на каждый запрос без кэширования.
- Кэш-слой: `app/services/cache_layer.py` (Redis или in-memory). Использовать для дорогих вычислений.
- FTS5-поиск: используется `tnved_fts` virtual table (runtime-индекс, создаётся при старте); Alembic **не управляет** FTS-таблицей (особенность SQLite FTS5 + Alembic).

## 11. Definition of Done (DoD)

Задача считается завершённой, если выполнены **все** пункты:

- [ ] RCA задокументирован (для баг-фиксов).
- [ ] Код изменён минимально — только по задаче.
- [ ] Тесты проходят (`pytest tests/ -v`).
- [ ] Ключевые эндпоинты проверены вручную через API.
- [ ] `git diff` просмотрен — нет случайных изменений.
- [ ] Миграции Alembic корректны (если менялась схема).
- [ ] `tsc --noEmit` зелёный (если менялся frontend).
- [ ] PR создан с описанием изменений и результатом тестов.
