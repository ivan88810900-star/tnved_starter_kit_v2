# Локальный классификатор ТН ВЭД (ONNX в процессе API)

Встроенный слой **до HTTP** `CUSTOM_CLASSIFIER_*` и LLM: при заданных путях к модели и меткам `POST /api/classify` сначала вызывает **onnxruntime**.

## Зависимость

```bash
cd customs-clear/backend
pip install -r requirements-ml.txt
```

## Переменные

| Переменная | Назначение |
|------------|------------|
| `ONNX_HS_CLASSIFIER_PATH` | Путь к файлу `.onnx` |
| `ONNX_HS_LABELS_PATH` | JSON-массив кодов ТН ВЭД (строки), **порядок = индекс класса** после `argmax` по логитам |
| `ONNX_HS_FEATURE_DIM` | Если у входа ONNX динамический размер (`None` в shape), задайте целое число — размерность вектора признаков |

## Контракт модели

1. **Вход:** один тензор `float32`, форма `[batch, D]`, обычно `batch=1`.
2. **Выход:** логиты `float32`, форма `[batch, K]`, `K` = число классов ≤ длины массива меток.

Текущая реализация в `app/services/onnx_hs_classifier.py` подаёт на вход **детерминированный хэш-вектор** текста описания размерности `D`. Это удобно для проверки интеграции, но **не даёт качества классификации**. Для боя экспортируйте модель, в которой уже есть векторизатор (или подавайте эмбеддинги из отдельного шага).

## Альтернатива: отдельный inference-сервис

Тот же JSON-ответ можно отдавать с внешнего процесса — см. **`INFERENCE_CLASSIFIER.md`** и `CUSTOM_CLASSIFIER_URL` (рекомендуется для Triton и тяжёлых моделей).

## Обучение и экспорт

См. **`docs/ML_TRAINING_PIPELINE.md`**. После обучения экспортируйте граф в ONNX (PyTorch `torch.onnx.export`, sklearn через `skl2onnx`, и т.д.) и согласуйте `D` и `K` с конфигом выше.
